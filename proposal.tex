\section{Project Topic Proposal}

Limited release clothing drops create a short, high intensity buying window where automated bots can acquire inventory faster than real shoppers. This reduces fairness for customers and increases operational burden for stores, especially when automated traffic overwhelms the purchase flow. Many commercial defenses rely on large scale signals such as device fingerprinting, third party intelligence feeds, and proprietary anti abuse infrastructure. Those systems are difficult to reproduce in a junior independent study. This project narrows the scope to a measurable, reproducible question: how much bot detection signal can be extracted from server side session logs and interaction timing, especially when the legitimate baseline comes from real human sessions instead of scripted simulations.

Prior work suggests that combining web log evidence with behavioral signals improves robustness against advanced automation. Iliou et al. show that pairing web logs with mouse behavioral biometrics can help detect advanced web bots that attempt to evade detection \cite{10.1145/3447815}. Challenge based mechanisms can also help, but they can increase friction and can be attacked, so the system design should be evaluated for usability tradeoffs \cite{10.1145/3672608.3707741}. Research on modeling bot sessions and arrival processes motivates a session level view of automation behavior that can be analyzed from aggregated request traces \cite{10.1145/3538969.3538994}. Building on these ideas, DropGuard Lite implements an interpretable, log based risk score and evaluates simple mitigation policies at add to cart and checkout.

\subsection*{What I will build}

The core software artifact is a minimal drop style web application with product browsing, cart management, and a simulated checkout flow. The application logs each request into an event table keyed by a random session identifier. A feature extraction pipeline aggregates events into per session features that capture speed, navigation path structure, and burst behavior. An interpretable risk score in the range $[0,1]$ is computed from weighted thresholds over these features. Finally, the application applies mitigation gates at sensitive actions such as add to cart and checkout, using a small set of policies such as allow, delay, or throttle or block.

\subsection*{Inputs and outputs}

Inputs include HTTP requests from two sources. The first source is real human sessions collected from volunteers using the site normally during controlled drop windows. The second source is automated bot sessions generated by a fast path checkout script that attempts product to cart to checkout with minimal delay. Outputs include the SQLite event log, a per session feature table, a per session risk score table, and experiment summaries that report bot share of inventory, real human checkout success rate, false positives, and friction measures such as added time or abandonment after mitigation.

\subsection*{Event logging schema}

Each request produces one event row with the following fields: event\_id, timestamp, session\_id, endpoint, method, status\_code, latency\_ms, and optional user\_agent and metadata\_json. The goal is to keep the logging schema simple enough to implement correctly, while still supporting meaningful per session analysis.

\subsection*{Real human sessions with permission}

A defining change in the revised proposal is that legitimate behavior is collected from real human sessions, not from a scripted human like simulator. Volunteers will see a consent screen before any telemetry is logged. The consent screen explains what is recorded, why it is recorded, and how it is stored. The system will not collect names, emails, addresses, payment information, or any direct identifiers. Checkout is simulated and records purchase success without processing payments, and participants will be instructed not to enter real personal data. Session identifiers are random and are not tied to identity. If the institution requires IRB or departmental review, I will follow the required process before collecting any human data.

\subsection*{Bot variants in scope}

To keep the project achievable while still realistic, I will implement one bot type in scope. The fast path checkout bot navigates product to cart to checkout with minimal delay. A stealth pacing bot that adds random delays and more normal navigation is a stretch goal if time permits.

\subsection*{Evaluation plan}

Evaluation will use repeated drop runs that mix real human sessions and bot sessions. Each run uses a fixed inventory count and a scheduled drop start time. I will compare mitigation thresholds and policies by measuring bot share of inventory, real human checkout success rate, false positive rate for humans who are delayed or blocked, and friction metrics such as added time to checkout and abandonment after mitigation. The intended outcome is a clear, reproducible tradeoff curve rather than a claim of universal bot prevention.

\section{Appendix: Feature List (Development Roadmap)}

Planned features for DropGuard Lite, in completion order:

\begin{enumerate}
  \item Implement product listing and product detail endpoints with fixed inventory and scheduled drop start time.
  \item Implement cart endpoints (add, remove, update quantity) tied to a session ID cookie.
  \item Implement simulated checkout that records purchase success and decrements inventory without collecting payment data.
  \item Implement an event logging layer that stores one row per request in SQLite using the defined schema.
  \item Implement a consent screen that requires explicit permission before any session telemetry is logged.
  \item Implement a data export script that outputs a clean session event table for analysis.
  \item Implement per session feature extraction (speed features, path features, burst features).
  \item Implement an interpretable risk scoring function that maps features to a score in $[0,1]$.
  \item Implement mitigation gates at add to cart and checkout (allow, delay, throttle or block) with configurable thresholds.
  \item Implement a fast checkout bot client script to generate automated traffic.
  \item Implement an experiment runner that mixes bot scripts with real human sessions and runs repeated trials.
  \item Implement evaluation metrics and produce summary tables and plots (bot inventory share, human success rate, false positives, friction).
\end{enumerate}

Stretch goals if time permits:

\begin{enumerate}
  \item Add a stealth pacing bot variant that randomizes timing and navigation to stress test the risk score.
  \item Add a read only monitoring page showing recent sessions, features, and risk scores.
  \item Replace fixed delays with a simple queue mechanism for high risk sessions.
\end{enumerate}

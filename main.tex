\section{Introduction}
Limited-release online stores create a buying environment where timing matters a lot. Many users try to purchase the same item at almost the exact same moment, and in that kind of setting even a small speed advantage can determine who gets through checkout first. Bots exploit this by moving through storefront flows more quickly and more consistently than most human shoppers can. While real shoppers can also act fast during a high-demand drop, their sessions usually show more variation, more hesitation, and more signs of ordinary browsing before they attempt to complete a purchase.

This is not only a technical problem. It is also a fairness problem. When the same kinds of automated sessions repeatedly win limited releases, legitimate buyers begin to feel that the system is unfair. Over time that damages trust in the storefront and in the release process itself. For large companies, this problem is often addressed with enterprise anti-bot systems that combine device signals, proprietary telemetry, behavioral data, and constantly updated detection rules \cite{10.1145/3447815}. Smaller sellers, however, usually do not have access to that kind of infrastructure. Even when stronger defenses are available, aggressive anti-bot controls can create a different problem by adding unnecessary friction for legitimate customers \cite{10.1145/3672608.3707741}.

This thesis examines whether a lighter and more explainable approach can still be useful. I present DropGuard Lite, a server-side prototype for bot risk scoring that relies on request logs and reconstructed session behavior rather than heavy client-side instrumentation or black-box detection models. The core idea is simple: instead of judging isolated requests one at a time, the system reconstructs a user session from backend activity, extracts behavioral features from that session, and then assigns a risk score based on patterns that appear suspicious in high-demand checkout settings.

The main research question of this thesis is whether ordinary backend data can support a practical and interpretable method for identifying suspicious fast-path checkout behavior. More specifically, I ask whether sessions that are likely bot-driven can be separated from legitimate human sessions using timing, navigation, and action-based patterns derived from server-observable events.

This project is intentionally modest in scope. I do not claim that one policy, one score, or one feature set can solve the broader web bot problem. I also do not claim that a lightweight scoring system can replace stronger multi-layer anti-abuse systems. Instead, I focus on a practical question: can a smaller storefront use the data it already collects to make better judgments about which sessions deserve closer attention during a competitive release?

\section{Related Work}
Prior work on web bot detection suggests that there is no single universal signal that cleanly separates automated traffic from legitimate user activity. Instead, different strands of research highlight different sources of evidence, each with its own tradeoffs. For this project, four themes are especially important: behavioral detection, server-log analysis, challenge-based defenses, and richer multi-signal detection systems.

One important theme is behavioral bot detection. Research in this area suggests that the way actions unfold over time can reveal more than simple counts of requests or static header information. Timing regularity, short transition windows, and repeated action patterns can all help indicate suspicious activity. Rahman and Tomar, for example, argue that behavior-oriented features can strengthen detection by focusing on how requests are generated rather than only on what resources are requested \cite{rahman2020new}. Tanaka and colleagues similarly show that combining user behavior with other request-level information can improve log-based bot detection \cite{tanaka2020bot}. The main takeaway from this line of work is that suspicious traffic often becomes easier to identify when viewed as a sequence rather than as a collection of independent requests.

A second theme is web log analysis and forensic investigation. This literature matters a great deal for my project because DropGuard Lite is built on the assumption that ordinary backend logs contain meaningful evidence. Stassopoulou and Dikaiakos showed early on that web robot detection could be framed probabilistically using server-side traces \cite{stassopoulou2009web}. Later work by Rahman and Tomar extended that idea by showing that request logs can support bot-oriented forensic analysis even without relying heavily on rich client instrumentation \cite{rahman2020forensic}. This is important because many smaller systems already collect backend logs for monitoring and debugging, which means session-level analysis can potentially be added without requiring a complete redesign of the storefront.

A third area includes challenge-based and rule-based defenses such as CAPTCHAs, queues, thresholds, and other friction mechanisms. These approaches remain common because they are relatively understandable and can block certain forms of abuse. However, they also create usability costs. A CAPTCHA or queue may successfully slow down bots, but it can also frustrate legitimate customers, especially during a time-sensitive release. Recent work such as Swiss Cheese CAPTCHA shows that challenge design continues to evolve, but it also reinforces the larger point that challenge systems involve a tradeoff between protection and user experience \cite{10.1145/3672608.3707741}. This tradeoff is one reason a risk-based approach is appealing. If suspicious sessions can be scored first, stronger friction can be applied more selectively.

A fourth theme is the rise of heavier anti-bot systems that combine multiple streams of evidence, such as device fingerprints, mouse dynamics, browser metadata, and broader telemetry. Iliou and coauthors, for instance, show that behavioral biometrics and log signals together can strengthen advanced web bot detection \cite{10.1145/3447815}. At the same time, later work on bot evasion using deep reinforcement learning makes clear that attackers can adapt to detection strategies over time \cite{10.1145/3538969.3538994}. This matters for my thesis in two ways. First, it shows why simple static rules are rarely enough on their own. Second, it highlights the practical gap between what large commercial systems can do and what smaller storefronts can realistically deploy.

The gap my thesis focuses on is therefore not whether richer systems can outperform simpler ones. They often can. The more relevant question is whether a smaller and more transparent system can still provide meaningful value in practice. Many storefronts need a baseline they can actually understand, inspect, and adjust. A lightweight system that reconstructs sessions, computes interpretable features, and supports human review may be less powerful than enterprise anti-bot infrastructure, but it may still be much more deployable.

My work differs from prior literature mainly in scope and deployment target. DropGuard Lite focuses on fast-path checkout behavior in limited-release storefronts, relies on backend activity rather than heavy telemetry, and treats explainability as a core requirement. Instead of presenting detection as an all-or-nothing classification problem, it frames output as a session-level risk score that can support different levels of intervention.

\section{Background}
Drop-based online commerce is built around scarcity, urgency, and competition. A store announces that a limited product will go live at a certain time, and a large number of users attempt to access the site almost simultaneously. Under these conditions, small differences in speed and consistency matter a great deal. A user who reaches checkout a few seconds earlier than another may secure the item, while the slower user misses it entirely. This creates a strong incentive to automate as much of the process as possible.

Fast-path bots are especially relevant in this environment. Rather than simulating broad exploration of the storefront, they are designed to move quickly through a narrow purchase path. In many cases, that means entering product pages immediately, adding an item to cart with minimal browsing, retrying critical endpoints aggressively, and reaching checkout with compressed timing. These patterns are not always exclusive to bots. Highly motivated human users may behave aggressively too, especially when they know a product is likely to sell out. Still, suspicious sessions often show a tighter, shallower, and more mechanically consistent path through the storefront.

This is why session reconstruction matters. Looking at one request at a time strips away the context needed to interpret behavior. A checkout request is perfectly normal in isolation. It becomes more or less suspicious only when placed inside a larger sequence. Did the session browse several pages before checkout, or did it jump into checkout almost immediately? Were there long pauses and normal variation between actions, or was the session tightly compressed? Did it retry cart or payment endpoints repeatedly in a narrow time window? These kinds of questions cannot be answered from individual requests alone \cite{rahman2020forensic,stassopoulou2009web}.

The choice to focus on server-side data is deliberate. It keeps the project grounded in signals that many systems already collect. Web applications often record timestamps, endpoints, HTTP methods, response codes, identifiers, and session-relevant metadata as part of routine monitoring. By using that information, the detection pipeline becomes easier to deploy and easier to explain. If a session is flagged, an operator can review the sequence of requests and understand which behaviors contributed to the score.

This approach also comes with clear limitations. Server-side logs do not capture everything, and they may be incomplete depending on how the storefront is implemented. As a result, the scope of DropGuard Lite is intentionally limited to backend-visible evidence and the suspicious behavior those signals can capture.

\section{Methodology}
The methodological goal of DropGuard Lite is to identify suspicious fast-path checkout behavior using reconstructed backend sessions. The system does not attempt to infer intent directly, and it does not rely on advanced fingerprinting or external threat intelligence. Instead, it treats risk scoring as a process of collecting session-level evidence and combining that evidence into a transparent assessment of suspicion.

The first step is event collection. The system records storefront activity related to browsing, cart actions, and checkout behavior. Each event includes basic information such as timestamp, event type, endpoint context, session identifier, and optional metadata. This information is captured through middleware and selected client-connected events, then stored for downstream analysis.

The second step is session reconstruction. Events are grouped into ordered timelines so that user behavior can be interpreted as a sequence. This step is essential because the meaning of most actions depends on what comes before and after them. A cart request that happens after several product views and a reasonable browsing interval may be unremarkable. The same request may look suspicious if it appears almost immediately in a very short and shallow session.

The third step is feature extraction. The features used in this project are designed around three main behavioral dimensions: timing, navigation, and action concentration.

Timing features capture speed and regularity, including session duration, inter-event spacing, and burstiness near critical endpoints. Navigation features capture browsing depth and coverage before purchase-critical activity. Action-based features capture retries, repeated bursts, and concentration around cart and checkout behavior.

The fourth step is risk scoring. In the current prototype, scoring is policy-based rather than learned through a black-box classifier. Each session is evaluated using a set of rules and weighted signals that reflect suspicious patterns. The output is not treated as a final accusation of bot activity. Instead, it is treated as a risk indicator that can support different responses depending on context.

This distinction between evidence and policy is important. The features describe what happened in the session. The thresholds describe how aggressively the system chooses to respond. Different storefronts may tolerate different levels of false positives. A store may decide to use the score only for monitoring, for queue prioritization, for selective friction, or for manual review. Keeping the evidence pipeline separate from the intervention policy makes those tradeoffs more visible and easier to adjust.

The methodology is scoped to fast-path checkout automation rather than every type of bot. A bot that scrapes content or mimics long browsing sessions may need a different feature set. Here, the main concern is suspicious speed, compressed flow, repeated critical actions, and minimal browsing depth during high-demand product releases.

\section{System Design and Implementation}
DropGuard Lite is implemented as a prototype using FastAPI, SQLite, and a lightweight React-based review interface. The architecture is intentionally simple because one of the goals of the project is to show that a usable bot-risk workflow can be built without a heavy infrastructure stack.

The system has four main layers: telemetry intake, storage, analytics, and review. The telemetry layer captures relevant storefront events as users move through browsing, cart, and checkout flows. Middleware records request-level activity, while selected application events add context where useful. These events are then written into storage.

The storage layer is designed around sessions and events. Event records preserve the sequence of actions over time. Session summaries are updated or inserted as activity arrives. Keeping these two views separate makes it easier to both reconstruct detailed traces and compute higher-level behavioral summaries.

The analytics layer transforms raw event logs into session-level outputs. It reconstructs ordered timelines, computes feature summaries, and prepares the data needed for risk scoring. This layer is where the project’s core logic lives. Instead of focusing on individual requests, it operates on the broader history of a session. That makes it possible to identify compressed checkout paths, repeated retries, and shallow navigation patterns that would be difficult to detect from isolated records.

The review layer exposes these outputs through a lightweight interface. Rather than presenting only a final score, the interface makes it possible to inspect event history, feature summaries, and session-level context. This is useful both for evaluation and for explainability. If a session appears suspicious, the reviewer can inspect the sequence directly and understand what contributed to the concern.

The processing pipeline moves through five stages. First, the system ingests incoming events. Second, it updates stored session state. Third, it reconstructs timelines from the recorded data. Fourth, it computes behavioral features. Fifth, it prepares the output for scoring and review. Structuring the system this way keeps the workflow modular and makes it easier to test individual components separately.

At the current stage of development, telemetry capture, session reconstruction, and feature extraction are implemented and working. The scoring layer is partially implemented, and the review workflow is already sufficient for inspecting suspicious and human-like sessions side by side. What remains less mature is the fully integrated mitigation layer. In other words, the project already supports detection-oriented analysis, but it is not yet a production-ready enforcement system.

This distinction matters because the thesis represents the project honestly. The core technical contribution is the session-based analysis pipeline and the risk-scoring framework built on top of it. DropGuard Lite is a research prototype, not a finished commercial anti-bot platform.

\section{Evaluation Setup}
The evaluation for this project is designed to answer a practical question: do reconstructed session traces contain enough information to help distinguish suspicious fast-path behavior from legitimate human checkout activity in a limited-release environment?

Because the project is still in progress, this section describes the evaluation design and the way the system will be assessed rather than presenting a completed experimental study. The goal at this stage is to show how the project will be evaluated once the remaining implementation and labeling work are finalized.

The planned evaluation compares two broad categories of sessions. The first category consists of legitimate human sessions collected in the project environment. These sessions represent real browsing and checkout behavior under ordinary or high-intensity conditions. The second category consists of suspicious or bot-like sessions represented through controlled automated traces and rule-labeled patterns. These traces are not meant to represent every possible bot strategy. Instead, they are meant to capture the narrower class of fast-path behavior that DropGuard Lite is designed to study.

The evaluation will focus on three main criteria.

First, I will evaluate separability. This asks whether suspicious sessions and human sessions show meaningful differences in the behavioral features computed by the system. If suspicious sessions consistently move faster, browse less, and concentrate their actions differently, then the feature set is capturing something useful.

Second, I will evaluate interpretability. A session-level scoring system is only useful in practice if operators can understand why a session appears risky. For that reason, the evaluation will include case-level walkthroughs of representative sessions across different risk bands. These walkthroughs will be used to show whether the system’s judgments line up with visible evidence in the trace.

Third, I will evaluate practical usefulness. Even if a feature set shows some statistical difference between groups, that alone does not make it operationally helpful. The more important question is whether the output could realistically support monitoring, manual review, queue placement, or selective friction in a small storefront environment.

The final quantitative version of the evaluation is expected to report precision, recall, false positive tendencies, and confusion patterns, along with ranked review of high-risk sessions. Sensitivity checks around thresholds and event windows are also planned, since release behavior can shift quickly during a drop. Label quality will be treated as a core validity issue rather than a background assumption.

Another part of the setup is practical review workflow. In addition to aggregate metrics, I plan to sample sessions from low, medium, and high risk bands and verify whether the underlying traces match the assigned level. This supports two goals at once: it checks whether the scoring logic is internally consistent, and it helps identify false positives early before stronger intervention policies are attached to the score.

\section{Results and Findings}
At the current stage of the project, the full quantitative evaluation has not yet been completed. For that reason, this section does not claim final performance results. Instead, it summarizes the patterns the current system is built to measure, the kinds of differences that are already visible in early traces, and the findings I expect the completed evaluation to test more formally.

Even in its current form, the prototype already supports telemetry capture, session reconstruction, and feature extraction. That makes it possible to inspect session histories directly and compare how suspicious traces differ from more typical human sessions. From these early comparisons, several patterns appear likely to matter.

The first expected finding is that suspicious fast-path sessions will move into cart and checkout more quickly than ordinary human sessions. In a limited-release setting, a bot designed to purchase as fast as possible would be expected to show very little buildup before purchase-oriented activity begins. Human users may also move quickly, especially during a competitive drop, but their sessions would still be expected to show more variation and less consistently compressed flow.

A second expected finding is that suspicious sessions will display tighter timing between requests, especially around purchase-critical endpoints. If the system works as intended, higher-risk sessions should show shorter inter-event gaps, more burst-like behavior, and more regular timing in sensitive parts of the checkout flow than legitimate sessions typically do.

A third expected finding is that suspicious sessions will show shallower navigation. Compared with human users, they would be expected to visit fewer parts of the storefront, spend less time exploring, and concentrate more of their behavior on endpoints that move them directly toward purchase. This will not be a perfect signal by itself, since some legitimate users may also arrive with a clear goal and browse very little, but it should still contribute useful evidence when interpreted alongside timing and retry behavior.

At the same time, I expect the completed evaluation to confirm an important limitation: no single feature will be strong enough on its own to separate suspicious behavior from legitimate human activity during a high-demand drop. The more realistic expectation is that combining timing, navigation, and action concentration will be more informative than any one feature alone.

An additional expected outcome is calibration value rather than only raw classification. Even if borderline sessions remain ambiguous, the system should still help separate clearly low-risk sessions from clearly high-risk sessions. That is useful in practice because operations teams can spend attention where it is most needed instead of treating all sessions the same.

\section{Analysis and Discussion}
Even before the final quantitative evaluation is complete, the structure of the project already suggests several important points about what the findings are likely to mean.

First, the thesis supports the idea that server-side behavioral evidence can still be useful, even without a large anti-bot infrastructure. Looking at behavior at the session level matters because suspicious activity is more visible in full sequences than in isolated requests.

Second, the project points toward a middle-ground approach to bot detection. Simpler methods are easier to deploy and inspect, even if they are likely to be less powerful than larger systems that combine many telemetry sources.

At the same time, the expected findings also come with clear limitations. Attackers can adapt over time and change the way their sessions look \cite{10.1145/3538969.3538994}. More advanced bots may mimic human pacing, introduce fake browsing depth, or spread behavior across identities. Legitimate users in high-pressure release windows may also resemble suspicious sessions on basic timing features. Because of that, I do not expect the final contribution of this thesis to be a claim that simple scoring solves bot abuse in general. A more reasonable conclusion is that it offers a useful first layer of judgment for a narrower problem setting.

Another likely implication is that selective intervention may be more appropriate than blanket friction. If the final system can identify a smaller set of higher-risk sessions, then a storefront could apply stronger defenses more selectively instead of challenging every user equally \cite{10.1145/3672608.3707741}. That possibility is one of the more practical reasons this kind of session-based system is appealing.

More broadly, the project suggests there is value in systems that are modest in scope but strong in transparency, especially for smaller teams that need a deployable and auditable baseline.

This also has implications for operational trust. A transparent scoring workflow gives teams a clearer way to explain decisions to internal reviewers and to improve policies over time. When a score appears wrong, the sequence can be inspected, the threshold can be adjusted, and the same case can be re-evaluated without rebuilding the whole system. That iterative loop is part of what makes a lightweight approach practical.

\section{Conclusion}
This thesis examines whether session-level analysis of backend activity can support practical bot-risk scoring in limited-release online storefronts. I present DropGuard Lite, a lightweight prototype that ingests storefront activity, reconstructs sessions, extracts interpretable behavioral features, and uses those features to support risk-informed review of suspicious checkout behavior.

Because the project is still being completed, the thesis does not claim final quantitative detection results. Instead, it argues that the design and early behavior of the system already support a promising direction for evaluation. The central expectation is that suspicious fast-path sessions will tend to appear tighter, shallower, and more concentrated around purchase-critical actions than ordinary human sessions, especially when timing, navigation, and retry behavior are considered together rather than in isolation.

If the final evaluation supports that expectation, then the main contribution of the project will be practical rather than absolute. DropGuard Lite would show that ordinary backend data can be organized into a meaningful and explainable workflow for bot-risk scoring, giving smaller storefronts a realistic first layer of defense and review. The likely value of the system is not that it will perfectly identify all automation, but that it can help operators make better judgments about which sessions deserve closer attention.

There are clear threats to validity. Label quality, overlap between aggressive human behavior and suspicious behavior, and differences across storefront contexts can all affect conclusions. Because the current system relies on server-observable behavior, it may miss advanced bots that imitate human pacing.

Future work includes broader validation across storefronts, more sequence-aware modeling with preserved interpretability, and better threshold adaptation across drop phases.

In the near term, the most important next step is completing the quantitative evaluation with careful label review. That will show more clearly where the current feature set is strong, where it is weak, and which threshold adjustments provide the best balance between detection value and user friction in this specific project setting.

Overall, this thesis argues that lightweight session-based bot-risk scoring is worth taking seriously as a practical first layer for smaller storefronts. It does not replace a full layered anti-abuse program, and it should not be presented as if it does. What it can do is provide useful evidence from data many systems already collect, helping operators make more informed and more explainable decisions about which sessions deserve closer attention.

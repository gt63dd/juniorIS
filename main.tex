\section{Introduction}
Limited-release online stores create a very specific kind of buying environment. A large number of people try to buy the same item at nearly the same time, often within a short release window. In that setting, even a small speed advantage can make a big difference. Bots take advantage of this by moving through product, cart, and checkout flows faster and more consistently than most human shoppers can. Real users may also move quickly during a drop, but their sessions usually show more variation in timing and more signs of normal browsing behavior.

This problem is not just about technical performance. When the same kinds of automated sessions keep winning, people begin to feel that the process is unfair. That can damage trust in the storefront and make the platform look unreliable. At the same time, many smaller sellers do not have access to large commercial anti-bot systems that rely on proprietary data, rich telemetry, and constantly updated models \cite{10.1145/3447815}. Overly aggressive defenses can also create a different problem by blocking legitimate buyers or making checkout harder than it needs to be \cite{10.1145/3672608.3707741}.

This thesis focuses on DropGuard Lite, a server-side approach to bot risk scoring built around request logs and session behavior. The system reconstructs sessions from backend activity, extracts behavioral features from those sessions, and assigns a risk score based on patterns that look suspicious. The main question is whether a lightweight and explainable method like this can help separate suspicious fast-path bot sessions from legitimate human sessions in limited-release storefront settings.

The rest of the paper follows that question step by step. It begins with background and related work, then explains the methodology behind the scoring approach, describes the system design and current implementation, outlines the evaluation setup, and presents draft findings based on the current state of the project. The paper closes with discussion, limitations, and directions for future work.

This project is intentionally modest in scope. It does not assume that one rule set or one model can solve every bot problem. Instead, it asks whether a simpler and more transparent approach can still be useful in practice, especially in settings where decisions need to be explained and improved over time. The goal is not perfect detection. The goal is to see whether common backend data can support better judgment about which sessions deserve closer attention.

\section{Related Works}

A growing body of research has examined methods for detecting automated web activity using behavioral and statistical techniques. While many approaches rely on complex machine learning systems, several studies demonstrate that carefully designed behavioral features can provide strong signals for identifying automated bots.

Rahman and Tomar \cite{rahman2020new} propose a feature engineering approach inspired by biostatistics that focuses on modeling human interaction patterns. Their work emphasizes the importance of features derived from real user behavior rather than simple request counts or network metadata. Examples include time spent on form fields, patterns of field navigation, and variations in user interaction timing. The authors argue that these signals are particularly valuable because they capture subtle aspects of human behavior that are difficult for automated scripts to reproduce consistently. This research provides a conceptual foundation for the feature extraction pipeline used in this project.

In related work, Rahman and Tomar \cite{rahman2020forensic} introduce a forensic framework for investigating bot-driven activity using web logs. Their framework highlights the importance of reconstructing user sessions from raw request logs and analyzing those sessions to detect abnormal behavior patterns. Rather than focusing solely on prevention mechanisms, the authors emphasize how structured analysis of logs can reveal the sequence of actions taken by automated programs. This approach supports the design of detection systems that operate at the session level rather than evaluating individual requests in isolation.

Other research has explored behavioral biometrics as a method for distinguishing automated traffic from legitimate users. Studies in this area analyze signals such as mouse movement patterns, cursor trajectories, and typing dynamics. These techniques can achieve high detection accuracy but often require extensive client-side instrumentation and data collection. While such methods are powerful, they may introduce privacy concerns and implementation complexity. As a result, many smaller systems instead rely on server-side signals that can be collected directly from HTTP logs.

Another line of work examines challenge-based defenses such as CAPTCHAs and rate limiting mechanisms. These defenses attempt to disrupt automated behavior by introducing tasks that are difficult for bots to solve. However, challenge-based approaches can create usability problems for legitimate users and may be circumvented by more sophisticated automation tools. Consequently, researchers increasingly recommend combining behavioral analysis with selective mitigation strategies rather than relying solely on explicit challenges.

The approach implemented in DropGuard Lite follows this trend toward interpretable behavioral detection. Instead of relying on heavy machine learning models or intrusive client-side instrumentation, the system analyzes server-side logs to identify suspicious patterns in request timing, navigation behavior, and action sequences. These features are aggregated into a simple risk score that can be used to trigger mitigation policies during critical stages of the purchase process.

By grounding the system design in existing research on behavioral bot detection and log-based forensic analysis, this project demonstrates how a lightweight, reproducible framework can still capture meaningful signals of automated purchasing behavior. The resulting system provides a practical testbed for evaluating how simple behavioral indicators can improve fairness during limited release product drops.

\section{Background}
Drop-based online commerce is built around scarcity and timing. Products are often released in small quantities, and many users try to buy them as soon as they become available. This environment creates a strong incentive for automation. Fast-path bot behavior in these settings often appears as short sessions that move quickly into cart and checkout, repeat certain actions several times, and show very little normal browsing before purchase attempts begin.

Because of that, individual requests usually do not tell the full story. What matters more is the order and timing of actions across an entire session. Session reconstruction makes it possible to group related requests together and view them as a sequence rather than as isolated events. Once that sequence is visible, patterns such as burstiness, shallow navigation, repeated retries, and compressed transitions become easier to identify \cite{rahman2020forensic,stassopoulou2009web}.

This thesis focuses on signals that can be observed from the server side. That choice is deliberate. It leaves out heavier approaches such as advanced fingerprinting or detailed behavioral biometrics, which may improve detection but also add more integration work and more governance concerns \cite{10.1145/3447815}. It also avoids treating challenge-based defenses as the main source of evidence, since tools like CAPTCHAs and queues can add friction for real users \cite{10.1145/3672608.3707741}. The question here is whether ordinary request data can still provide useful information for bot detection.

This scope also makes the system easier to think about from a deployment standpoint. Most web applications already collect request logs for debugging, monitoring, or analytics. Using that existing data lowers the barrier to implementation and makes the results easier to audit. When a session is flagged, reviewers can trace that outcome back to concrete request behavior instead of relying on a model that is difficult to explain. That matters in high-demand release settings, where users may challenge outcomes and operators need a clear reason for why a session was flagged or allowed.
